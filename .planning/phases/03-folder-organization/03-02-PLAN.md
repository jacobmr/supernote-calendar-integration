<task type="auto">
  <name>Task 1: Create FolderOrganizer service</name>
  <files>src/services/folder-organizer.ts, tests/folder-organizer.test.ts</files>
  <action>
1. Create `src/services/folder-organizer.ts`:
   - Import SupernoteAPIClient, FolderMappingStore, MeetingData
   - Class `FolderOrganizer`:
     - Constructor takes `supernoteClient: SupernoteAPIClient`, `mappingStore: FolderMappingStore`
     - Private `basePath: string = "Calendar"` (root folder in Supernote for all meeting folders)
     - `async ensureBaseStructure(): Promise<{ recurringFolderId: number, adHocFolderId: number }>`:
       - Creates `Calendar/Recurring` and `Calendar/Ad-Hoc` using `supernoteClient.createFolderPath()`
       - Returns folder IDs for both. Cache these IDs on the instance after first call.
     - `sanitizeFolderName(name: string): string`:
       - Reuse same logic as `sanitizeNotebookName` in integration.ts (replace invalid chars, limit to 100 chars, trim)
       - Do NOT import from integration.ts — copy the 3-line function to avoid coupling
     - `async createFolderForMeeting(meeting: MeetingData): Promise<void>`:
       - If `mappingStore.hasFolderForMeeting(meeting.id)` → skip (already has folder)
       - If meeting.isRecurring:
         - Check `mappingStore.hasFolderForRecurringEvent(meeting.recurringEventId!)` → if yes, just add instance mapping pointing to same folder, return
         - Create folder: `Calendar/Recurring/{sanitized meeting title}/` using createFolderPath
         - Save to mappingStore as both a recurring folder and an instance mapping
       - If NOT recurring:
         - Create folder: `Calendar/Ad-Hoc/` (meetings share this folder — individual note files will go here in Phase 4)
         - Save mapping with the Ad-Hoc folder ID
       - Log all folder creation actions
     - `async processNewMeetings(meetings: MeetingData[]): Promise<{ created: number, skipped: number }>`:
       - Call ensureBaseStructure() once
       - Loop through meetings, call createFolderForMeeting for each
       - Add 500ms delay between Supernote API calls (rate limiting per API findings)
       - Return counts of created vs skipped
       - Wrap each meeting in try/catch — log errors but continue processing remaining meetings

2. Create `tests/folder-organizer.test.ts`:
   - Mock SupernoteAPIClient (createFolderPath returns predictable IDs)
   - Mock FolderMappingStore
   - Test: ensureBaseStructure creates Calendar/Recurring and Calendar/Ad-Hoc paths
   - Test: recurring meeting creates folder under Recurring/[Meeting-Name]
   - Test: second instance of same recurring meeting reuses existing folder (no duplicate creation)
   - Test: ad-hoc meeting maps to Ad-Hoc folder (no per-meeting subfolder)
   - Test: meeting with existing folder mapping is skipped
   - Test: sanitizeFolderName handles special characters
   - Test: processNewMeetings continues after individual meeting error

Use jest.useFakeTimers() or mock the delay to avoid slow tests.

<task type="auto">
  <name>Task 2: Integrate FolderOrganizer into scheduler</name>
  <files>src/index-scheduler.ts</files>
  <action>
1. In `src/index-scheduler.ts`:
   - Import FolderOrganizer, FolderMappingStore, SupernoteAPIClient
   - After the existing change detection logic (where newMeetings/changedMeetings/cancelledMeetings are determined):
     - If there are new meetings AND Supernote credentials are available (SUPERNOTE_EMAIL + SUPERNOTE_PASSWORD env vars):
       - Authenticate SupernoteAPIClient
       - Create FolderMappingStore instance
       - Create FolderOrganizer instance
       - Call `processNewMeetings(newMeetings)` passing the MeetingData[] for new meetings
       - Log results: "Folders created: X, skipped: Y"
     - If Supernote credentials are NOT available:
       - Log warning: "Supernote credentials not configured — skipping folder creation"
       - Do NOT throw — scheduler should continue running for change detection even without Supernote
   - Update the scheduler-status.json write to include folder creation counts:
     - Add `foldersCreated` and `foldersSkipped` to the status output
   - Wrap the entire folder creation block in try/catch — folder creation failures should NOT crash the scheduler or prevent state saving

2. Do NOT modify the change detection logic or state saving. Folder creation is additive — it runs after change detection succeeds.

Note: The scheduler currently receives PersistedMeeting[] from StateManager, but FolderOrganizer needs MeetingData[] (which includes isRecurring). The meeting detector already produces MeetingData[] — pass these through to the folder organizer before they get converted to PersistedMeeting[]. This may require light refactoring of the scheduler flow to retain the MeetingData[] from the detector alongside the PersistedMeeting[] used for state.

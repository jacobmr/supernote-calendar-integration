<task type="auto">
  <name>Task 1: Create Caddyfile and production Docker Compose</name>
  <files>Caddyfile, docker-compose.prod.yml</files>
  <action>
1. Create `Caddyfile` in project root:
   ```
   supernote.salundo.com {
       reverse_proxy app:3000
   }
   ```
   That's it — Caddy auto-provisions TLS via Let's Encrypt when it can reach the domain.

2. Create `docker-compose.prod.yml` (production override, NOT replacing dev compose):

   ```yaml
   services:
     caddy:
       image: caddy:2-alpine
       ports:
         - "80:80"
         - "443:443"
       volumes:
         - ./Caddyfile:/etc/caddy/Caddyfile:ro
         - caddy_data:/data
         - caddy_config:/config
       depends_on:
         app:
           condition: service_healthy
       restart: unless-stopped

     app:
       build: .
       expose:
         - "3000"
       volumes:
         - app_data:/app/data
         - ./.env:/app/.env:ro
       restart: unless-stopped
       healthcheck:
         test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
         interval: 30s
         timeout: 10s
         retries: 3
         start_period: 15s

   volumes:
     caddy_data:
     caddy_config:
     app_data:
   ```

   Key differences from dev compose:
   - App uses `expose` not `ports` (only Caddy faces the network)
   - Caddy sidecar handles TLS termination
   - Named volumes for persistence (not bind mounts for data — more robust)
   - Caddy waits for app health check before starting

   Do NOT bind-mount app data to ./data in production — use named Docker volumes for reliability. The .env bind mount is fine since it's read-only config.
   </action>

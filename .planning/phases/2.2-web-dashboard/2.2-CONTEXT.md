# Phase 2.2: Web Dashboard & Setup Interface - Context

**Gathered:** 2026-03-12
**Status:** Ready for planning

---

<vision>
## How This Should Work

When your wife opens the dashboard for the first time, she encounters a **setup wizard**:

1. Simple form asking for her Supernote email and password
2. Button to trigger Google OAuth (she clicks, authenticates, comes back)
3. Confirmation that setup is complete

On **subsequent visits**, she lands on the dashboard showing:

- Last scheduler run summary: how many meetings were **new**, **changed**, and **cancelled**
- Clear, readable layout (no charts, no jargon)
- A button to manually trigger the scheduler if she wants to test it

The entire interface should feel **simple and calm** — not overwhelming. Everything she needs to know in one glance.

</vision>

<essential>
## What Must Be Nailed

**The dashboard must be simple and readable.** No confusing charts or jargon.

If we get one thing right, it's this: when she opens the page after setup, she instantly understands what's happening with her calendar. "Oh, 3 new meetings, 1 was cancelled, got it."

</essential>

<boundaries>
## What's Out of Scope

- **No real-time updates** — Page refresh is fine to see new data
- **No past history/analytics** — Just show today's state, not historical tracking
- **No modification of schedules** — Dashboard is read-only (except manual trigger)
- **No mobile optimization** — Desktop/tablet focused
- **No fancy UI frameworks** — Clean, minimal HTML/CSS/JS

**Does include:**

- Simple password protection (so random internet people don't stumble on it)

</boundaries>

<specifics>
## Specific Ideas

**Setup flow:**

- Form fields: Supernote email, Supernote password
- "Sign in with Google" button (triggers OAuth in browser, captures refresh token)
- Simple validation (don't let her submit empty fields)
- Success message when both steps are done

**Dashboard display:**

- Show something like: "Last run: 3 minutes ago"
- "New meetings: 5, Changed: 2, Cancelled: 1"
- Manual trigger button (with confirmation: "Run scheduler now?")
- Maybe a simple "Status: Running" / "Status: Idle" indicator

**Styling:**

- Clean HTML, no Bootstrap or fancy CSS frameworks
- Dark mode appreciated (already using that elsewhere in the app)
- Text-heavy, not icon-heavy (she shouldn't have to guess what icons mean)

</specifics>

<notes>
## Additional Context

- **Security concern:** Simple password protection is enough; we're not building enterprise auth
- **First-time friction:** Setup must be foolproof — she shouldn't get stuck entering credentials or confused by OAuth
- **No WebSocket/real-time:** Page refresh is totally acceptable
- **Manual trigger:** Nice-to-have for testing, but not critical to first version
- **Error handling:** If credentials fail validation, clear message about what went wrong

The goal is a **non-technical person** (your wife) should be able to open this URL, complete setup once, then check status whenever she wants. No confusion, no jargon.

</notes>
